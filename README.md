## The Problem

Corporate carbon accounting is broken. The [BCG Climate Survey 2025](https://bcg.com) found that only **7% of large companies** comprehensively measure emissions across all scopes — down from 10% two years prior. The gap between sustainability *spending* and actual *measurement* is widening.

Two root causes:

| Problem | Scale |
|---|---|
| **Scope 3 is unmeasured** — supply chain emissions are 80–90% of most companies' footprint, but depend on hundreds of suppliers who may not track their own emissions | Affects ~93% of large companies |
| **Claims are unverifiable** — sustainability reports are self-authored with no standardized audit requirement equivalent to financial reporting | No public tool existed to cross-reference them |

Existing tools (Watershed, Persefoni) cost **$50K+/year** and require dedicated teams to operate. CarbonLens is free, open, and upload-and-go.

## Solution: Two Modes, One Platform

```
┌─────────────────────────────────────────────────────────────┐
│                        CarbonLens                           │
│                                                             │
│   🥕  MEASURE MODE              🪵  VERIFY MODE            │
│   ─────────────────            ────────────────             │
│   Upload procurement CSV       Enter any company name       │
│   → Scope 3 breakdown          → Transparency Score 0–100   │
│   → Supplier hotspots          → Forensic findings          │
│   → Ranked recommendations     → EPA cross-reference        │
│                                                             │
│   "The carrot"                 "The stick"                  │
│   Helps companies improve      Holds companies accountable  │
└─────────────────────────────────────────────────────────────┘
```

### Measure Mode
Upload any procurement CSV or Excel file. Four AI agents parse the data, classify each line item into GHG Protocol Scope 3 categories, calculate emissions using verified EPA and DEFRA databases, and generate ranked recommendations with estimated reduction potential.

**Sample output — Nexus Electronics (55 line items):**
- Total Scope 3: **12,787 tCO2e**
- Top emitter: Taiwan Semiconductor Partners (steel + semiconductor procurement)
- Key recommendation: Switch Apex Steel from BOF to EAF process → ~60% reduction on that line

### Verify Mode
Enter any company name. Five AI agents build a company profile, extract every quantitative claim from their sustainability report, gather independent EPA facility-level data, forensically cross-reference the two, and generate a Transparency Score.

**Sample output — ExxonMobil:**
- Transparency Score: **48/100 — Low Transparency**
- Key finding: Net-zero target excludes Scope 3 (83% of total footprint)
- EPA gap: 45+ million tCO2e difference between reported and independently verified emissions

## Architecture

### The Core Design Principle

> **LLMs handle understanding. Code handles math.**

No emission number is ever generated by an LLM. Every number traces back to a published government database and an explicit multiplication formula. This is the anti-hallucination architecture that makes outputs auditable.

```
User Request
     │
     ▼
POST /api/verify  ──► returns job_id immediately (< 200ms)
     │
     ▼
Background asyncio task ──► 5-agent pipeline runs here
     │
     ├── Agent 1: Company Intelligence  [Claude Sonnet]
     ├── Agent 2: Report Extraction     [Claude Sonnet]
     ├── Agent 3: Independent Data      [Claude Sonnet + Python/SQLite]
     ├── Agent 4: Cross-Reference       [Claude Opus  ← reasoning-intensive]
     └── Agent 5: Report Generation     [Claude Sonnet]
     │
     ▼
GET /api/verify/{job_id}  ◄── Frontend polls every 1.5s
     │
     ▼
{ status, current_agent, agents: [...], result: {...} }
```

The **fire-and-forget + polling** pattern is used throughout. The POST endpoint accepts work instantly. A background task does the work. The GET endpoint reports progress. This prevents browser timeouts on 30–60 second pipelines and enables the real-time agent progress UI.

## The Agent Pipelines

### Verify Mode — 5 Agents

Every agent follows the same pattern: pull context from shared state → build prompt → call Claude API → parse JSON response → write output back to state.

```python
state = {"company_name": "ExxonMobil"}

state = company_intelligence_agent(state)   # Agent 1 — Claude Sonnet
# + company_profile: {revenue, sector, facility_count, epa_ids, ...}

state = report_extraction_agent(state)      # Agent 2 — Claude Sonnet
# + claims_extracted: [{claim_text, scope, metric_type, value, ...}, ...]

state = independent_data_agent(state)       # Agent 3 — Sonnet + SQLite
# + independent_data: {ghgrp_total, benchmark_estimate, news_items, ...}

state = cross_reference_agent(state)        # Agent 4 — Claude OPUS
# + transparency_score: 48
# + findings: [{type, severity, evidence, ...}, ...]

state = report_generation_agent(state)      # Agent 5 — Claude Sonnet
# + executive_summary, narrative, methodology_statement
```

**Why Opus for Agent 4?** Agent 4 simultaneously reasons across three independent data sources (sustainability report, EPA facility data, industry benchmarks), catches subtle accounting techniques (intensity framing, boundary exclusions, baseline manipulation), and must distinguish "company is wrong" from "company uses a different methodology." That forensic complexity justifies the cost premium.

**All Claude calls use `temperature=0`** — structured data extraction requires deterministic output. Downstream agents parse the JSON; format variations between runs would cause crashes.

### Measure Mode — 4 Agents

```python
state = {"raw_input": file_content}

state = data_ingestion_agent(state)         # Agent 1 — Claude Sonnet
# Handles messy CSVs: normalizes columns, strips currency symbols,
# excludes subtotals/blank rows, infers missing supplier names

state = category_classification_agent(state) # Agent 2 — Claude Sonnet
# Maps each line to GHG Protocol Scope 3 category (1–15)
# Assigns EEIO sector code or DEFRA lookup key
# Prefers activity-based (DEFRA) when quantity+unit data exists

state = emission_factor_calculate(state)    # Agent 3 — PURE PYTHON, ZERO LLM CALLS
# SQLite lookup → deterministic multiplication → citation string

state = analysis_recommendation_agent(state) # Agent 4 — Claude Sonnet
# Hotspot identification, supplier ranking, specific actionable recommendations
```

### The Anti-Hallucination Guarantee

Agent 3 in Measure Mode contains zero LLM calls. Every emission number is:

```
source line item  ×  government database factor  =  output
─────────────────────────────────────────────────────────
$1,270,000 USD   ×  0.98 kgCO2e/USD (EPA EEIO 2021,  =  1,244,600 kgCO2e
(Apex Steel)        sector 331110)
```

The citation string `"1,270,000 USD × 0.98 kgCO2e/USD = 1,244,600 kgCO2e [EPA_EEIO 2021, sector 331110]"` is stored alongside every result. An auditor can verify the number without running the software.

**Five anti-hallucination rules enforced by design:**

1. No LLM ever generates an emission factor number
2. All arithmetic is Python code — never prompted
3. Every number includes: factor source, year, method, confidence tag
4. Full citation chain stored per line item
5. Cross-reference agent is prompted: *"ONLY flag findings with concrete evidence. Never speculate."*

## Transparency Scoring Algorithm

The 0–100 score is built from four weighted sub-scores. The score is **not** generated by Claude — it is calculated deterministically in Python from the findings Agent 4 returns.

```
┌─────────────────┬────────────────┬──────────────────┬─────────────┐
│  Completeness   │  Consistency   │  Verifiability   │  Ambition   │
│    25 points    │    30 points   │    25 points     │  20 points  │
├─────────────────┼────────────────┼──────────────────┼─────────────┤
│ Reports all     │ Numbers match  │ % of claims with │ Science-    │
│ scopes? Full    │ EPA data?      │ independent      │ based       │
│ boundary        │ Internally     │ cross-reference  │ targets?    │
│ disclosure?     │ consistent?    │ available        │ Real        │
│ Absolute +      │                │                  │ progress?   │
│ intensity?      │                │                  │             │
└─────────────────┴────────────────┴──────────────────┴─────────────┘
```

**Why Consistency has the highest weight (30%):** It's the only sub-score that can't be gamed. Either your reported number matches independently recorded EPA facility data, or it doesn't. Completeness can be improved by publishing more data. Ambition can be improved by announcing bigger targets. Consistency requires your numbers to actually be correct.

**Score bands:**

```
80–100  ██████████  Green   High Transparency
60–79   ████████    Teal    Moderate Transparency
40–59   ██████      Amber   Low Transparency
0–39    ████        Red     Very Low Transparency
```

Finding severities drive deductions:

```python
# Example: EPA_DISCREPANCY HIGH deducts 15 from Consistency
deductions = {
    "EPA_DISCREPANCY":       {"HIGH": 15, "MEDIUM": 9,  "LOW": 4},
    "INTENSITY_VS_ABSOLUTE": {"HIGH": 10, "MEDIUM": 6,  "LOW": 3},
    "MISSING_SCOPE":         {"HIGH": 10, "MEDIUM": 6,  "LOW": 3},
    "TARGET_WITHOUT_PLAN":   {"HIGH": 10, "MEDIUM": 6,  "LOW": 3},
    "BASELINE_MANIPULATION": {"HIGH": 8,  "MEDIUM": 5,  "LOW": 2},
    "BOUNDARY_EXCLUSION":    {"HIGH": 7,  "MEDIUM": 4,  "LOW": 2},
}
```

Each sub-score is independently clamped to `max(0, score)` — severe findings in one dimension cannot bleed into another.

## Database Schema

```sql
-- Spend-based emission factors (EPA EEIO) — ~36 sectors
-- Used by Measure Mode Agent 3
CREATE TABLE eeio_factors (
    id                INTEGER PRIMARY KEY,
    sector_code       TEXT,    -- NAICS code e.g. "324110"
    sector_name       TEXT,    -- "Petroleum Refineries"
    ef_kgco2e_per_usd REAL,    -- e.g. 1.28 (range: 0.04 – 1.82)
    ef_year           INTEGER,
    source            TEXT DEFAULT 'EPA_EEIO'
);

-- Activity-based factors (UK DEFRA) — ~22 categories
-- Preferred when quantity+unit data exists (more accurate than spend-based)
CREATE TABLE defra_factors (
    id          INTEGER PRIMARY KEY,
    category    TEXT,    -- "Business travel - flights"
    subcategory TEXT,    -- "Long-haul, economy"
    unit        TEXT,    -- "kgCO2e per passenger-km"
    ef_value    REAL,
    ef_year     INTEGER,
    source      TEXT DEFAULT 'DEFRA'
);

-- EPA facility-level emissions (GHGRP) — ~250 records
-- Crown jewel of Verify Mode — independent data to cross-reference claims
CREATE TABLE ghgrp_facilities (
    id                     INTEGER PRIMARY KEY,
    facility_name          TEXT,
    parent_company         TEXT,
    city                   TEXT,
    state                  TEXT,
    latitude               REAL,
    longitude              REAL,
    naics_code             TEXT,
    industry_sector        TEXT,
    reporting_year         INTEGER,
    total_emissions_mtco2e REAL,
    co2_emissions          REAL,
    methane_emissions      REAL,
    n2o_emissions          REAL
);

-- Sector average emission intensities — 15 sectors
-- Powers the estimation engine in both modes
CREATE TABLE industry_benchmarks (
    id                              INTEGER PRIMARY KEY,
    sector                          TEXT,
    avg_intensity_tco2e_per_m_revenue REAL,
    median_intensity                REAL,
    p25_intensity                   REAL,
    p75_intensity                   REAL,
    sample_size                     INTEGER,
    source                          TEXT,
    year                            INTEGER
);

-- Completed job results — survives server restarts
CREATE TABLE jobs (
    job_type     TEXT,
    job_id       TEXT PRIMARY KEY,
    company_name TEXT,
    result       TEXT,    -- full result JSON serialized as string
    created_at   TEXT
);
```

**Note on DEFRA vs EEIO selection logic:**
```
Has quantity + unit?  →  DEFRA (activity-based, more accurate)
Dollar amount only?   →  EEIO  (spend-based, directional)
No factor match?      →  Flag as "no_emission_factor", DO NOT estimate
```

## Getting Started

### Prerequisites

- Python 3.11+
- Node.js 18+
- An [Anthropic API key](https://console.anthropic.com)
- A [Google Gemini API key](https://aistudio.google.com)

### Local Development

**1. Clone the repository**
```bash
git clone https://github.com/RastinAghighi/CarbonLens.git
cd CarbonLens
```

**2. Backend setup**
```bash
cd backend

# Create and activate virtual environment
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Set environment variables
cp .env.example .env
# Add your ANTHROPIC_API_KEY and GEMINI_API_KEY to .env

# Initialize the database (creates tables + seeds all emission factor data)
python scripts/init_database.py
python scripts/load_ghgrp_data.py

# Start the backend
uvicorn main:app --reload --port 8080
```

**3. Frontend setup**
```bash
cd frontend

npm install

# Set the backend URL
echo "VITE_API_URL=http://localhost:8080" > .env.local

npm run dev
# → http://localhost:5173
```

**4. Try it**
- **Measure:** Upload `sample_data/sample_procurement_nexus_electronics.csv`
- **Verify:** Enter `ExxonMobil`, `Shell`, `Amazon`, or `Google`

### Environment Variables

| Variable | Required | Description |
|---|---|---|
| `ANTHROPIC_API_KEY` | ✅ | Anthropic Claude API key (all agents) |
| `GEMINI_API_KEY` | ✅ | Google Gemini API key (chat endpoint) |
| `VITE_API_URL` | ✅ Frontend | Backend URL (defaults to Cloud Run URL) |

### Docker (Production)

```bash
cd backend
docker build -t carbonlens-backend .
docker run -p 8080:8080 \
  -e ANTHROPIC_API_KEY=your_key \
  -e GEMINI_API_KEY=your_key \
  carbonlens-backend
```

The Dockerfile automatically:
1. Installs system and Python dependencies
2. Runs `init_database.py` to seed emission factor tables
3. Runs `load_ghgrp_data.py` to load facility records
4. Starts Uvicorn on port 8080
